## Context

`taiwanway` 網站目前已存在團購功能，但其在首頁上的可見性不高。本次設計是為了將現有團購功能的精選內容呈現在首頁，提高用戶發現與參與度，同時確保數據能即時、準確地來自 Google Sheets。我們需要整合前端 UI、數據獲取策略和數據同步機制。

## Goals / Non-Goals

**Goals:**
*   在 `taiwanway` 網站首頁上，實作一個動態展示活躍團購活動的區塊。
*   確保首頁展示的團購數據是最新且可靠的，並能與 Google Sheets 保持同步。
*   優化數據獲取機制，確保首頁加載性能不受影響。
*   提供清晰的團購活動連結，引導用戶至詳情頁。

**Non-Goals:**
*   本次設計不包含對現有團購活動創建、編輯、管理等後台功能的重構。
*   不涉及對 Google Sheets 團購數據結構的根本性修改（除非是為了同步便利而進行微調）。
*   不實作一套全新的團購管理系統，而是基於現有功能進行強化。
*   不處理多個 Google Sheets 來源的團購數據整合。

## Decisions

### Decision 1: 首頁團購數據獲取方式
*   **選項 A (優選)**：**API 數據接口**。在後端提供一個專為首頁設計的 API 端點，該端點會從數據庫中獲取活躍的、精簡的團購數據。前端通過調用此 API 來獲取數據。
    *   **理由**：
        *   **性能**：後端可以針對首頁需求進行優化查詢，只返回必要字段和數量，減少數據傳輸量。
        *   **分離關注點**：前後端職責分離，後端負責數據處理，前端負責 UI 呈現。
        *   **安全性**：前端不直接訪問數據庫或 Google Sheets。
        *   **緩存**：後端 API 可以實作數據緩存機制。
*   **選項 B**：**直接在前端處理**。前端直接從某個靜態 JSON 文件或直接讀取 Google Sheets 數據。
    *   **理由**：部署簡單。
    *   **缺點**：
        *   安全性：直接讀取 Google Sheets 可能洩露憑證。
        *   性能：前端可能需要加載更多不必要的數據。
        *   複雜性：數據處理邏輯混入前端。

### Decision 2: Google Sheet 數據同步機制
*   **選項 A (優選)**：**後端定時任務（Cron Job）+ 手動觸發**。建立一個後端服務，負責定時（例如每 15-30 分鐘）從 Google Sheets 抓取最新團購數據，並更新到網站的數據庫中。同時，提供一個後台管理介面或 API 端點，允許管理員手動觸發同步。
    *   **理由**：
        *   **自動化**：確保數據準確性和即時性。
        *   **可靠性**：後端服務處理數據同步，減少前端依賴。
        *   **可控性**：管理員可手動干預。
*   **選項 B**：**前端即時同步**。每次首頁加載時，前端都嘗試從 Google Sheets 獲取數據。
    *   **缺點**：
        *   性能：可能導致首頁加載延遲。
        *   成本：Google Sheets API 調用次數過多。會造成Google服務計價。
        *   安全性：前端直接訪問 Google Sheets API 存在風險。

### Decision 3: 首頁團購展示組件
*   **選項 A (優選)**：**創建新的獨立 React 組件**。例如 `HomepageGroupBuySection.js`，負責獲取數據並渲染團購列表。
    *   **理由**：
        *   **模組化**：易於維護、測試和重用。
        *   **關注點分離**：將首頁邏輯與團購展示邏輯分離。
*   **選項 B**：直接修改現有的首頁組件。
    *   **缺點**：可能導致組件過於龐大，難以管理。

## Risks / Trade-offs

*   **風險**：Google Sheets API 配額限制 → **緩解**：優化同步頻率，實作後端緩存，並監控 API 使用情況。
*   **風險**：Google Sheets 數據結構變更 → **緩解**：同步邏輯需健壯，能識別並處理結構不匹配，並有日誌和警報機制。
*   **風險**：首頁性能下降 → **緩解**：API 端點優化查詢，前端組件加載優化，使用 Skeleton Screen 或 Lazy Loading。
*   **權衡**：增加後端複雜度以換取更好的性能和數據可靠性。

## Migration Plan

1.  **開發階段**：在開發環境中逐步實作前端組件、後端 API 端點和數據同步服務。
2.  **測試階段**：在測試環境中進行全面的單元測試、整合測試和端對端測試，特別是數據同步的準確性和首頁展示的性能。
3.  **部署階段**：
    *   首先部署後端數據同步服務和新的 API 端點。
    *   確認數據同步正常運行後，再部署前端更改（包含新的首頁團購區塊）。
4.  **回滾策略**：
    *   若部署後端服務出現問題，可回滾到前一版本。
    *   若部署前端後首頁出現問題，可回滾前端部署。
    *   前後端部署應可獨立回滾。
